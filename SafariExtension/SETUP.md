# YTDL Bridge — implementation notes

Safari Web Extension + a companion macOS app that hands off the current tab's
URL to the local `ytdl` CLI and saves the download to `~/Downloads/`.

## Layout

```
SafariExtension/
├── Local.xcconfig.sample                # copy to Local.xcconfig with your values
├── YTDLBridge.xcodeproj/
├── YTDLBridge/                          # container app (App target, Sandbox OFF)
│   ├── AppDelegate.swift                # receives ytdlbridge:// and spawns ytdl
│   ├── ViewController.swift             # info page shown after launch (Xcode template)
│   ├── Info.plist                       # registers CFBundleURLTypes = ytdlbridge
│   ├── Assets.xcassets/
│   ├── Base.lproj/Main.storyboard
│   └── Resources/{Main.html, Script.js, Style.css, Icon.png}
├── YTDLBridge Extension/                # Safari Web Extension (appex, Sandbox ON)
│   ├── SafariWebExtensionHandler.swift  # turns popup messages into ytdlbridge:// URLs
│   ├── Info.plist                       # NSExtension dict only
│   └── Resources/
│       ├── manifest.json                # MV3, activeTab / nativeMessaging / notifications
│       ├── background.js                # bridge between popup and the native handler
│       ├── popup.html / popup.js        # "video" and "audio only" buttons
│       └── icons/{16,32,48,128}.png
├── YTDLBridgeTests/  YTDLBridgeUITests/ # generated by Xcode (unused)
├── scripts/build_release.sh             # produces a notarized .dmg
└── SETUP.md
```

Bundle identifiers:
- App: `org.cathand.YTDLBridge`
- Extension: `org.cathand.YTDLBridge.Extension`

## Architecture

```
Safari toolbar button
  → popup.html/js
    → browser.runtime.sendMessage
      → background.js: sendNativeMessage
        → SafariWebExtensionHandler (Sandbox ON)
          → NSWorkspace.shared.open("ytdlbridge://download?url=...")
            → YTDLBridge.app (Sandbox OFF)
              → application(_:open:) or handleGetURL
                → Process.run("/path/to/.venv/bin/ytdl -d ~/Downloads <URL>")
                  → ~/Downloads/<title>.mp4
```

## Xcode settings that matter (things not to touch blindly)

Deviations from the stock "Safari Extension App" template. Some settings are
brittle — read this before changing them.

### App target (YTDLBridge)

- **Remove App Sandbox in Signing & Capabilities** (× button in Xcode UI). This
  is what lets `Process.run` spawn `ytdl`.
- **Add CFBundleURLTypes**: `YTDLBridge/Info.plist` registers the `ytdlbridge`
  scheme, and Build Settings sets `INFOPLIST_FILE = YTDLBridge/Info.plist`.
- Hardened Runtime = ON (required for notarization).
- Do **not** use `applicationScriptsDirectory` / `NSUserUnixTask` — with the
  sandbox off, `Process.run` is enough.

### Extension target (YTDLBridge Extension)

- **Keep App Sandbox ON.** Safari only discovers sandboxed Web Extensions;
  turning it off makes the extension disappear from Safari's Extensions list.
- Keep `Info.plist` minimal (`NSExtension` dict only). Writing extra `CFBundle*`
  keys by hand conflicts with `GENERATE_INFOPLIST_FILE = YES` and breaks the
  extension.
- `MACOSX_DEPLOYMENT_TARGET` must match the app's. The template default (10.14)
  isn't enough — `SFExtensionMessageKey` requires 11.0+.

### Traps I hit and avoided

- **Adding `CODE_SIGN_ENTITLEMENTS` to `.pbxproj` by hand**: Safari stops
  discovering the extension.
- **Hand-written `Info.plist` with `CFBundleIdentifier` / `CFBundlePackageType`**:
  conflicts with the auto-generated one and breaks the extension.
- **Disabling App Sandbox on the extension**: extension vanishes from Safari.
- **Signing the debug build with Developer ID Application**: in some states
  Launch Services gets confused and the extension isn't discovered. Stick with
  Apple Development (Automatic) for local dev.

## First-time setup

Assumes Xcode + paid Apple Developer Program membership + `ytdl` installed in a
venv + `deno` installed.

1. **Copy `SafariExtension/Local.xcconfig.sample` to `SafariExtension/Local.xcconfig`
   and fill it in:**
   ```
   DEVELOPMENT_TEAM = XXXXXXXXXX               # your Apple Team ID
   YTDL_BIN_PATH    = /path/to/ytdl/.venv/bin/ytdl
   ```
   `Local.xcconfig` is gitignored so your personal values don't leak.

2. Open `YTDLBridge.xcodeproj` in Xcode (both targets pick up the xcconfig).
3. Press **⌘R** to build and launch YTDLBridge.
4. Enable **YTDL Bridge** in **Safari > Settings > Extensions** and allow it on
   every website.
5. On first launch, macOS asks for Full Disk Access (needed to read Safari's
   cookies) — grant it.
6. Open a YouTube tab, click the YTDL Bridge toolbar icon, choose **Download
   video**.

## Release build (notarized `.dmg`)

```bash
cd SafariExtension
export TEAM_ID=XXXXXXXXXX          # your Apple Team ID (same as Local.xcconfig)
export NOTARY_PROFILE=AC_NOTARY    # name you passed to xcrun notarytool store-credentials
./scripts/build_release.sh
# → build/YTDLBridge.dmg
```

First-time notary credential setup:

```bash
xcrun notarytool store-credentials AC_NOTARY \
  --apple-id "<your-apple-id>" \
  --team-id  "$TEAM_ID" \
  --password "<app-specific-password>"
```

## Debugging

- **Extension popup / background**: Safari > Develop menu > "Web Extension
  Background Content" > YTDL Bridge opens the Web Inspector.
- **Container app logs**: `log stream --predicate 'subsystem == "org.cathand.YTDLBridge"' --info`
- **Extension logs**: `log stream --predicate 'subsystem == "org.cathand.YTDLBridge.Extension"' --info`

## Limitations

- Destination is hardcoded to `~/Downloads/`.
- No quality picker beyond video / audio-only.
- The `ytdl` binary path is set via `Local.xcconfig` (`YTDL_BIN_PATH`) and baked
  into `Info.plist` at build time; update the xcconfig and rebuild if the venv
  moves.
